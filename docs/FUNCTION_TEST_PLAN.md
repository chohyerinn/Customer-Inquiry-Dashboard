## 1. 멀티채널 문의 수집 (Multi-channel Inquiry Collection)

### 1-1. 웹폼 제출 — 정상 경로

| 단계 | 내용 |
|------|------|
| 입력 | `POST /inquiries` Body: `{ "name": "홍길동", "contact": "test-user@example.com", "category": "tech", "urgency": "medium", "body": "로그인이 안 됩니다." }` |
| 처리 중 | FastAPI가 요청 수신 → InquiryDTO로 정규화 → 파이프라인 진입 |
| 기대 결과 | HTTP 201, `{ "inquiry_id": "INQ-0001", "status": "received", "assigned_counselor": "..." }` |
| 구현 확인 필요 | `POST /inquiries` 엔드포인트, InquiryDTO 스키마 |

### 1-2. 웹폼 제출 — 오류 상태

| 시나리오 | 입력 | 기대 결과 |
|----------|------|-----------|
| 빈 body 필드 | `body: ""` | HTTP 400, `"error": "inquiry body is required"` |
| 필수 필드 누락 (contact 없음) | `contact` 키 자체 생략 | HTTP 422 Unprocessable Entity |
| 지원하지 않는 category | `"category": "unknown"` | HTTP 400, `"error": "invalid category"` |
| 페이로드 크기 초과 | body 필드 > 10,000자 | HTTP 413 또는 400 |
| 구현 확인 필요 | 입력 유효성 검사 로직 전반 | |

### 1-3. 웹폼 제출 — 빈 상태

| 시나리오 | 기대 결과 |
|----------|-----------|
| `GET /inquiries` — DB에 데이터 없음 | HTTP 200, `{ "inquiries": [], "total": 0 }` |
| 대시보드 첫 접속 (문의 0건) | "No inquiries yet" 메시지 + 일러스트 표시 |
| 구현 확인 필요 | 빈 리스트 응답 처리 및 프론트엔드 EmptyState 컴포넌트 |

| 전체 | DB 저장 + 대시보드 업데이트 | **< 2초** |
| 구현 확인 필요 | 전체 파이프라인 통합 테스트 | |

### Scenario B — PII 포함 긴급 문의 (이메일, 긴급도 HIGH)

| 단계 | 기대 결과 |
|------|-----------|
| IMAP 수집 | 이메일 → InquiryDTO 변환 |
| PII 탐지 | 가상 전화번호 마스킹, `raw_text` CLOVA에 미전달 검증 |
| 분류 | `category: "billing"`, `urgency: "high"` |
| 초안 생성 | CLOVA에 `masked_text`만 전달 |
| 배정 | 최소 부하 상담원 배정 |
| 알림 | Telegram 채널 메시지 수신 |
| 전체 처리 시간 | **< 5초** |
| 구현 확인 필요 | CLOVA 요청 로그에서 원문 미포함 검증 필수 |

### Scenario C — CLOVA API 장애

| 단계 | 기대 결과 |
|------|-----------|
