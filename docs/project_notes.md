# Project Notes

## Why I built this

고객 문의 자동화에서 제일 위험한 부분은 “AI가 답변을 만들어준다” 자체보다, 개인정보가 외부 API로 나가거나 긴급 문의가 일반 문의처럼 처리되는 상황이라고 생각했다. 그래서 이 프로젝트는 상담원을 대체하는 챗봇이 아니라, 문의를 모으고 분류하고 개인정보를 걸러낸 뒤 상담원이 확인할 수 있는 초안을 만드는 흐름으로 잡았다.

처음부터 완전 자동 응답 시스템을 만들기보다는, 실제 운영에서 필요한 PII 차단, 긴급도 분류, 폴백 응답, 큐 배정, 비용 확인을 한 파이프라인으로 묶어보는 것이 목표였다.

## What was difficult

가장 어려웠던 부분은 “LLM 호출 전에 무엇을 막아야 하는지”를 정하는 것이었다. 사용자가 입력한 문의에는 이메일, 전화번호, 주문번호, 계좌 같은 정보가 섞일 수 있다. 이런 텍스트를 그대로 외부 AI API에 보내면 안 되기 때문에, 답변 생성보다 앞단의 PII 게이트가 더 중요했다.

또 하나 어려웠던 부분은 장애 상황을 정상 흐름에 넣는 것이었다. CLOVA API가 실패하거나 timeout이 나도 고객 문의 처리는 멈추면 안 된다. 그래서 서킷브레이커와 템플릿 응답을 넣어서, AI 응답이 없어도 최소한의 운영 흐름은 유지되게 했다.

## Issues I ran into

### 1. PII가 포함된 문의를 어떻게 처리할지 애매했다

단순히 마스킹만 하고 AI에 보내면 되는지 고민했지만, 개인정보가 포함된 문의는 마스킹 누락 위험이 있다. 그래서 PII가 감지되면 외부 AI 호출을 하지 않고 템플릿 응답으로 처리하는 방향으로 잡았다.

### 2. 분류 모델보다 규칙 기반 분류가 더 방어 가능했다

처음에는 LLM으로 카테고리와 긴급도를 모두 판단하게 할 수도 있었지만, 운영 관점에서는 왜 HIGH로 분류됐는지 설명할 수 있어야 했다. 그래서 키워드 기반 규칙으로 카테고리와 긴급도를 먼저 나누고, AI는 PII 없는 일반 문의의 답변 초안에만 사용했다.

### 3. CLOVA 장애 상황을 고려해야 했다

AI API 호출은 항상 성공한다고 가정하기 어렵다. timeout이나 오류가 반복되면 계속 외부 API를 호출하는 것도 비용과 안정성 측면에서 좋지 않다. 그래서 실패가 반복되면 서킷브레이커를 열고, 일정 시간 동안 템플릿 응답으로 폴백하도록 했다.

### 4. 대시보드는 예쁘게 만드는 것보다 운영 지표가 먼저였다

처음에는 화면을 더 꾸미고 싶었지만, 이 프로젝트에서는 채널별 문의 수, 긴급도, PII 감지, 큐 상태, 비용, 서킷 상태 같은 운영 지표가 더 중요했다. 그래서 대시보드는 “보기 좋은 UI”보다 현재 상태를 빨리 확인하는 쪽에 맞췄다.

## How I fixed them

- 문의 접수 후 가장 먼저 PII 탐지와 마스킹을 수행했다.
- PII 포함 문의는 외부 LLM으로 보내지 않고 템플릿 응답을 사용했다.
- 키워드 기반 카테고리/긴급도 분류를 만들어 분류 이유를 추적하기 쉽게 했다.
- CLOVA 호출 실패 시 템플릿 응답으로 폴백하고, 반복 실패 시 서킷브레이커를 열도록 했다.
- 긴급 문의와 PII 문의는 일반 큐가 아니라 전담/전문 큐로 배정했다.
- 비용 집계와 Telegram 알림을 넣어 운영 중 확인해야 할 이벤트를 남겼다.

## What I learned

AI 고객응대 프로젝트에서 중요한 것은 답변 생성만이 아니었다. 입력 검증, 개인정보 차단, 장애 폴백, 상담원 검토 흐름이 있어야 실제 업무에 가까워진다는 걸 배웠다.

또 LLM을 모든 판단에 쓰는 것보다, 설명 가능한 규칙과 LLM을 나눠 쓰는 편이 더 안전할 수 있다는 것도 느꼈다. 특히 개인정보나 긴급도처럼 운영 리스크가 큰 부분은 규칙과 로그가 남는 방식이 더 방어 가능했다.

## What I would improve next

- 실제 상담 데이터가 있다면 오탐/미탐 케이스를 모아 PII 규칙을 더 개선하고 싶다.
- 문의 분류 결과에 대해 상담원이 수정한 기록을 저장해서 규칙 개선에 쓰고 싶다.
- 대시보드에 일별 추이와 fallback 비율을 추가하고 싶다.
- 테스트와 CI를 추가해서 PII 차단, 폴백, 큐 배정이 깨지지 않게 만들고 싶다.
